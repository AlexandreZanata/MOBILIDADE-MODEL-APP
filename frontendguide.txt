Guia Completo do Frontend para Corridas
(Passageiro e Motorista)
Este documento explica, passo a passo, como integrar o frontend (web ou mobile) com a API de
corridas. Você encontrará os fluxos completos para passageiros e motoristas, payloads de requisição,
respostas esperadas, uso de websockets, gerenciamento de estados e dicas de UX.
0. Arquitetura agnóstica (frontend independente)
1. Separação clara: o backend expõe apenas APIs REST e eventos websocket. O frontend pode ser
qualquer stack (React, Vue, Flutter, Swift, Android, etc.) desde que fale HTTP/JSON.
2. Camada de acesso: crie um módulo de serviços (por exemplo, TripApiClient ) que encapsule
fetch , axios , retrofit ou qualquer HTTP client. Evite importar helpers ou libs específicas
do Laravel/PHP.
3. Ambientes configuráveis: parametrize API_BASE_URL , chaves Pusher/Soketi e feature flags
via arquivos .env do app ou variáveis nativas das lojas.
4. BFF opcional: se precisar mesclar dados ou cachear respostas, crie um Backend-for-Frontend
separado. Este guia presume consumo direto da API, mas nada impede a existência de um BFF.
5. Versionamento e telemetria: inclua cabeçalhos como X-Client-Platform / X-Client-
Version para facilitar observabilidade no backend.
1. Pré-requisitos e Setup
1. Autenticação JWT
Todas as rotas abaixo exigem Authorization: Bearer <token> .
Certifique-se de ter implementado o fluxo de login/refresh para passageiros
( /passengers/login ) e motoristas ( /auth/login / fluxo de aprovação).
2. Configurações do ambiente
Base URL (ex.: https://api.suaapp.com/api ).
Headers padrão:Authorization: Bearer <token>
Accept: application/json
Content-Type: application/json
3. Websockets (Soketi/Pusher compatível)
Configure o frontend com Laravel Echo ou Pusher JS apontando para as credenciais
definidas no .env ( PUSHER_APP_KEY , host, porta).
Os canais privados usados são:
private-drivers.{driverId} – eventos destinados ao motorista.
private-passengers.{passengerId} – eventos destinados ao passageiro.
4. Estados locais
Mantenha no frontend um store (Redux, Zustand, Vuex, etc.) com as informações da trip
corrente: status , driver , passenger , estimated_fare ,
matched_driver_id , assignment_expires_at , etc.
2. Fluxo do Passageiro
2.1 Estimar corrida
1. Coletar origem (lat/lng atual) e destino (selecionado no mapa).
2. Chamar:
POST /trips/estimate
Body:
{
"origin": { "lat": -23.55052, "lng": -46.63331 },
"destination": { "lat": -23.56111, "lng": -46.65655 },
"trip_category_id": "<uuid>",
"surge_multiplier": 1.2 // opcional
}
3. Resposta: distance_km , duration_seconds , final_fare , minimum_fare ,
category_multiplier , etc.
4. Exibir esses dados para o usuário (valor estimado, tempo aproximado).
2.2 Seleção do método de pagamento
1. Liste os métodos disponíveis (ex.: crédito, débito, dinheiro) com dados vindos do seu backend.
2. Se o método exige bandeira (cartão), mostre as bandeiras possíveis.3. Salve o payment_method_id e payment_brand_id para enviar no passo seguinte.
2.3 Solicitar corrida
1. Após o passageiro confirmar:
POST /trips
Body:
{
"origin": { "lat": ..., "lng": ... },
"destination": { "lat": ..., "lng": ... },
"trip_category_id": "<uuid>",
"payment_method_id": "<uuid>",
"payment_brand_id": "<uuid>" // se necessário
}
⚠️ IMPORTANTE - Segurança do Preço: O backend SEMPRE recalcula o preço no
momento da criação da corrida, independentemente do valor estimado mostrado ao
usuário. O frontend NÃO deve enviar valores de preço, surge_multiplier ou
qualquer outro parâmetro que possa influenciar o cálculo. O surge_multiplier é
calculado automaticamente pelo backend baseado em demanda real. Isso garante que
usuários maliciosos não possam manipular o preço da corrida.
2. Resposta:
trip_id
status = REQUESTED
estimated_fare (calculado pelo backend), driver_id (null inicialmente),
expires_at
3. Frontend deve:
Guardar o trip_id .
Exibir tela de "procurando motoristas".
Iniciar polling ou (preferencial) escutar evento TripStatusChanged em private-
passengers.{passengerId} .
2.5 Listar categorias com preço estimado
Use POST /trips/categories/estimate enviando apenas origem/destino (e
opcionalmente surge_multiplier para simulação visual).A resposta devolve um array de categorias ativas, cada uma com o valor estimado
( final_fare , distance_km , duration_seconds ).
Exiba os cards de seleção (Comfort, Premium etc.) com esses valores antes de enviar POST
/trips .
Nota: O surge_multiplier neste endpoint é apenas para fins de estimativa visual. Na
criação real da corrida ( POST /trips ), o backend recalcula o preço automaticamente e ignora
qualquer valor enviado pelo frontend.
2.6 Acompanhar matching
1. Assim que o backend atribuir um motorista, dois eventos podem ocorrer:
DriverTripAssigned (o app do motorista recebe primeiro).
TripStatusChanged (passageiro recebe notificação de status=DRIVER_ASSIGNED ,
com driver_id e timestamps).
2. Quando receber TripStatusChanged :
Atualize UI com as informações do motorista (veículo, nome, rating). Obtenha do seu
backend via GET /drivers/profile/{driverId} ou rota equivalente.
Mostre contagem regressiva caso queira indicar tempo de chegada.
2.7 Visualizar detalhes da corrida
Use GET /trips/{tripId} para obter snapshot do status, motorista e metadata.
Recomenda-se combinar GET inicial + websocket para manter sincronizado.
2.8 Cancelar corrida
POST /trips/{tripId}/cancel
Body opcional: { "reason": "Mudança de planos" }
Regras no backend:
Passageiro só cancela enquanto a corrida não estiver COMPLETED .
Se o motorista já estiver chegando, informe o usuário sobre possíveis penalidades (dependendo
das regras do negócio).
2.9 Avaliar motorista
Após status=COMPLETED :POST /trips/{tripId}/rating
{
"rating": 5,
"comment": "Excelente experiência"
}
Só aceita uma avaliação por corrida e requer status finalizado.
Atualize sua UI para impedir múltiplos envios.
3. Fluxo do Motorista
3.1 Disponibilidade e localização
1. Ativar disponibilidade (sugestão: POST /drivers/location/availability com {
"is_available": true } ).
2. Enviar geolocalização continuamente: PUT /drivers/location
{
"latitude": -23.55,
"longitude": -46.63,
"heading": 120,
"speed": 30,
"is_available": true
}
Recomendado a cada 5s (limite da API: 12 req/min).
Atualize is_available=false quando estiver offline/deslogado.
3.2 Receber corridas via websocket
1. Conectar ao Echo/Pusher:
Echo.private(`drivers.${driverId}`)
.listen('.TripAssigned', payload => { ... })
.listen('.TripStatusChanged', payload => { ... });
2. Ao receber TripAssigned :
Exibir modal com origem/destino, valor estimado, assignment_expires_at .
Disparar contagem regressiva de 15s.
3.3 Aceitar corridaPOST /drivers/me/trips/{tripId}/accept
Sucesso → status DRIVER_ARRIVING , lock liberado.
Atualize UI para “a caminho do passageiro”.
3.4 Atualizar status da corrida
Use POST /trips/{tripId}/status com corpo:
{ "status": "DRIVER_ARRIVED" }

4. Websockets e Estado em Tempo Real (Agnóstico)
Esta seção explica como conectar e escutar eventos WebSocket de forma agnóstica, funcionando
tanto para React Native, Flutter, Web (React/Vue), quanto qualquer outra plataforma.
4.1 Configuração BaseO backend usa Soketi (compatível com Pusher Protocol v7), que funciona com qualquer cliente que
suporte o protocolo Pusher.
Variáveis de ambiente necessárias:
PUSHER_APP_ID=<seu_app_id>
PUSHER_APP_KEY=<sua_app_key>
PUSHER_APP_SECRET=<seu_app_secret>
PUSHER_HOST=<host_do_soketi> (ex: localhost ou seu_dominio.com)
PUSHER_PORT=6001 (ou a porta configurada)
PUSHER_SCHEME=http (ou https)
4.2 Autenticação de Canais Privados
Canais privados ( private-drivers.* e private-passengers.* ) requerem autenticação. O
Laravel Broadcasting expõe um endpoint para isso:
Endpoint de autenticação:
POST /broadcasting/auth
Headers:
Authorization: Bearer <seu_jwt_token>
Accept: application/json
Content-Type: application/json
Body:
{
"socket_id": "<socket_id_fornecido_pelo_cliente>",
"channel_name": "private-drivers.{driverId}" // ou "private-passengers.
{passengerId}"
}
Resposta (200 OK):
{
"auth": "<token_de_autenticacao_do_canal>"
}

4.5 Tratamento de Erros e Reconexão
Erros comuns:
401 Unauthorized no /broadcasting/auth : Token JWT expirado ou inválido. Renove o token
e tente novamente.
403 Forbidden no /broadcasting/auth : Usuário não tem permissão para acessar o canal
(ex.: motorista tentando acessar canal de passageiro).
Conexão perdida: Implemente reconexão automática. A maioria dos clientes Pusher faz isso
automaticamente.
Reconexão e sincronização:
Sempre que o app voltar do background ou reconectar, faça um GET /trips/{tripId}
(passageiro) ou GET /drivers/me/trips/active (motorista) para garantir que o estado
local está alinhado.
Caso perca eventos enquanto offline, o polling garante consistência.
Implemente um mecanismo de "heartbeat" para detectar conexões mortas e reconectar
automaticamente.4.6 Exemplo Completo (React Native com Pusher)
import { useEffect, useState } from 'react';
import Pusher from 'pusher-js/react-native';
function useTripWebSocket(userId, userType) {
const [tripStatus, setTripStatus] = useState(null);
const [pendingTrip, setPendingTrip] = useState(null);
useEffect(() => {
const channelName = userType === 'motorista'
? `private-drivers.${userId}`
: `private-passengers.${userId}`;
const pusher = new Pusher(PUSHER_APP_KEY, {
cluster: 'mt1',
authEndpoint: `${API_BASE_URL}/broadcasting/auth`,
auth: {
headers: {
'Authorization': `Bearer ${accessToken}`,
'Accept': 'application/json',
},
},
wsHost: PUSHER_HOST,
wsPort: PUSHER_PORT,
enabledTransports: ['ws', 'wss'],
});
const channel = pusher.subscribe(channelName);
if (userType === 'motorista') {
channel.bind('TripAssigned', (data) => {
setPendingTrip({
tripId: data.trip_id,
origin: data.origin,
destination: data.destination,
estimatedFare: data.estimated_fare,
expiresAt: data.assignment_expires_at,
});
});
}
channel.bind('TripStatusChanged', (data) => {setTripStatus({
tripId: data.trip_id,
status: data.status,
driverSnapshot: data.driver_snapshot,
timestamps: data.timestamps,
});
});
return () => {
channel.unbind_all();
channel.unsubscribe();
pusher.disconnect();
};
}, [userId, userType]);
return { tripStatus, pendingTrip };
}

5.2 Estados “Edge”
Pas. tenta criar nova corrida com uma ativa → mostrar opção “retomar corrida atual” usando
GET /trips/{id} .
Motorista fora do raio ao tentar marcar “cheguei” → alertar para chegar mais perto (backend
valida 100m).Timeout de aceite → UI precisa lidar com retorno ao estado de “procurando corridas”
automaticamente.
5.3 Logs e telemetria
Logue sempre o trip_id , status e ação executada (para debug). Isso ajuda a sincronizar
com logs do backend caso haja qualquer divergência.

7. Testes recomendados
1. Fluxo completo passageiro: criar corrida, receber matching, acompanhar até completar e avaliar.
2. Timeout de motorista: Simular motorista que não aceita – verificar reatribuição e status
EXPIRED .
3. Cancelamento passageiro: Cancelar em diferentes fases (antes do matching, depois do
matching, durante a corrida).
4. Cancelamento motorista: Usar status=CANCELED_BY_DRIVER com reason .
5. Eventos em tempo real: Desconectar e reconectar WebSocket para garantir que o app sincroniza
via fallback (GET).
6. Validação de proximidade: Tentar marcar “cheguei” longe do destino – confirmar mensagem de
erro amigável.
7. Avaliação duplicada: Tentar enviar nota duas vezes e garantir que o backend retorna 422.
8. Resumo
Sempre armazene trip_id e acompanhe via TripStatusChanged + GET .
Atualize a localização do motorista frequentemente para garantir que ele participe do matching.
Use o tarifador (estimativa) antes da solicitação para informar o passageiro.Implemente fluxos de cancelamento e avaliação obedecendo as regras de status.
Assine os canais corretos ( drivers.* , passengers.* ) para uma UX em tempo real.
Valide as respostas e erros para orientar o usuário em cada etapa.
Seguindo este guia, o frontend terá todos os dados e eventos necessários para oferecer uma
experiência completa e confiável tanto para motoristas quanto para passageiros no ecossistema de
corridas.
