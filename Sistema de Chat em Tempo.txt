1. Visão Geral
O sistema de chat permite comunicação em tempo real entre motoristas e passageiros durante uma
corrida ativa. O chat é habilitado automaticamente quando uma corrida é aceita pelo motorista e
permanece disponível até a conclusão ou cancelamento da corrida.
Características Principais
Tempo Real: WebSocket para comunicação bidirecional instantânea
Garantia de Entrega: Mensagens são persistidas e entregues mesmo se o destinatário estiver
offlineStatus de Entrega: Confirmação de entrega e leitura
Badge Count: Contador de mensagens não lidas por corrida
Status Online: Rastreamento de presença em tempo real
Rate Limiting: Proteção contra spam e abuso
Retry Automático: Tentativas automáticas com exponential backoff
DLQ: Dead Letter Queue para mensagens que falharam após múltiplas tentativas
Graceful Degradation: Fallback para long polling quando WebSocket não está disponível
Lazy Loading: Carregamento sob demanda de histórico de mensagens
Cache Redis: Mensagens recentes em cache para performance

Fluxo de Mensagem
1. Envio: Cliente envia mensagem via WebSocket ou REST API
2. Validação: Rate limiting e validação de conteúdo
3. Persistência: Mensagem salva no PostgreSQL
4. Publicação: Mensagem publicada no RabbitMQ
5. Entrega: RabbitMQ entrega ao destinatário via WebSocket (se online) ou armazena para entrega
posterior
6. Confirmação: Destinatário confirma recebimento
7. Leitura: Destinatário marca como lida
8. Cache: Mensagens recentes armazenadas no Redis
3. Protocolos e Tecnologias
WebSocket (Primário)
Protocolo: ws:// ou wss:// (TLS)
Endpoints:
Motoristas: /ws/drivers/chat?token={JWT}
Passageiros: /ws/passengers/chat?token={JWT}
Formato: JSON
Reconexão: Automática com exponential backoff
Long Polling (Fallback)
Protocolo: HTTP/HTTPS
Endpoint: GET /v1/chat/messages/poll?rideId={rideId}&timeout=30Timeout: 30 segundos (configurável)
Uso: Quando WebSocket não está disponível
RabbitMQ
Exchange: chat.exchange (Direct)
Fila Principal: chat.messages
Fila Retry: chat.messages.retry (TTL: 60s)
DLQ: chat.messages.dlq
Exchange Pub/Sub: chat.delivery.exchange (Topic)
Redis
Cache de Mensagens: chat:messages:{rideId}:recent (TTL: 1h)
Status Online: chat:online:{userId} (TTL: 5min)
Rate Limiting: chat:ratelimit:{userId} (TTL: 1min)
Badge Count: chat:unread:{userId}:{rideId} (TTL: 24h)

8. Rate Limiting
Limites
Mensagens por minuto: 30
Burst: 10 mensagens em 1 segundo
Janela de tempo: 60 segundos
Implementação
Usa Redis com sliding window:
String key = "chat:ratelimit:" + userId;
Long count = redisTemplate.opsForValue().increment(key);
if (count == 1) {
redisTemplate.expire(key, 60, TimeUnit.SECONDS);
}
if (count > 30) {
throw new RateLimitExceededException();
}

Persistência
Todas as mensagens são persistidas no PostgreSQL antes de serem publicadas no RabbitMQ:
1. Mensagem salva com status SENDING
2. Publicada no RabbitMQ
3. Se destinatário estiver online, entrega imediata via WebSocket
4. Status atualizado para DELIVERED
5. Se destinatário estiver offline, mensagem fica pendente
6. Quando destinatário se conecta, recebe todas as mensagens pendentes
Confirmação de Entrega
Destinatário confirma recebimento via WebSocket
Status atualizado para DELIVERED
deliveredAt preenchido
Confirmação de Leitura
Destinatário marca mensagem como lida
Status atualizado para READ
readAt preenchido
Badge count decrementado
10. Retry e DLQ
Retry Logic
1. Tentativa 1: Mensagem enviada imediatamente
2. Falha: Mensagem vai para chat.messages.retry (TTL: 60s)
3. Tentativa 2: Após 60s, mensagem retorna para chat.messages
4. Falha: Vai para retry novamente
5. Tentativa 3: Última tentativa
6. Falha: Mensagem vai para DLQ
Exponential Backoff (Reconexão WebSocket)let reconnectDelay = 1000; // 1 segundo
const maxDelay = 60000; // 60 segundos

}

11. Status de Entrega e Leitura
Status Possíveis
SENDING: Mensagem sendo enviada (ainda não confirmada)
DELIVERED: Mensagem entregue ao destinatário
READ: Mensagem lida pelo destinatário
FAILED: Falha ao entregar (irá para DLQ)
Fluxo
1. Mensagem criada → SENDING
2. Publicada no RabbitMQ
3. Se destinatário online → entrega via WebSocket → DELIVERED
4. Destinatário lê → READ
Timestamps
createdAt : Quando mensagem foi criada
deliveredAt : Quando mensagem foi entreguereadAt : Quando mensagem foi lida
